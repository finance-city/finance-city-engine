## 수정된 버그

### 1. Invalid RenderPipeline — WGSL 유니포미티 위반

**증상**: 브라우저 콘솔에 `[Invalid RenderPipeline "Building Instancing Pipeline"] is invalid due to a previous error` 출력, 건물 전체가 렌더링되지 않음

**원인**: `shaders/building.wgsl`의 `fs_main` 내부에서 `textureSample()` 호출이 비-균일(non-uniform) 제어 흐름 (`if (isClose && isVertical)`) 안에 있었습니다. WebGPU WGSL 유니포미티 분석 규칙에 의해 `textureSample`은 모든 invocation이 도달하는 균일한 분기 안에서만 사용할 수 있습니다. 이를 위반하면 셰이더 모듈 컴파일 실패 → 파이프라인이 Invalid 상태로 마킹됩니다.

**수정**: 명시적 LOD를 사용하는 `textureSampleLevel(..., 0.0)`으로 교체. 명시적 LOD 함수는 묵시적 편미분(ddx/ddy)을 계산하지 않으므로 비-균일 분기 안에서도 유효합니다.

```wgsl
// 이전 (유니포미티 위반)
if (isClose && isVertical) {
    let rawN = textureSample(buildingNormalArray, anisoSampler, scaledUV, layerIdx).rgb;
}

// 이후 (명시적 LOD — 비-균일 분기 허용)
if (isClose && isVertical) {
    let rawN = textureSampleLevel(buildingNormalArray, anisoSampler, scaledUV, layerIdx, 0.0).rgb;
}
```

**수정 파일**: `shaders/building.wgsl`

### 2. 건물 전체 검정색 — Albedo 수식 오류

**증상**: 모든 건물이 완전한 검정색으로 렌더링됨

**원인**: Albedo 합성 수식 `mix(texAlbedo, texAlbedo * tint, 0.35)`에서 `texAlbedo`가 0에 가까울 때 tint 색상도 함께 소거되었습니다. 텍스처 로딩이 실패하거나 텍스처가 매우 어두울 경우 결과가 검정색이 됩니다.

**수정**: tint를 독립 항으로 분리하여 텍스처 상태와 무관하게 색상이 출력되도록 변경.

```wgsl
// 이전 — 텍스처가 0이면 tint도 소거됨
let albedoBase = mix(texAlbedoLinear, texAlbedoLinear * tintLinear, 0.35);

// 이후 — tint가 독립적으로 살아있음
let albedoBase = mix(texAlbedoLinear, tintLinear, 0.35);
```

**수정 파일**: `shaders/building.wgsl`

### 3. 건물 전체 검정색 — TextureManager 바인딩 누락

**증상**: TextureManager 초기화 실패 시 바인딩 7(albedo array), 8(normal array), 11(aniso sampler)이 bind group에서 누락되어 draw call 자체가 거부됨

**원인**: `generate()` 실패 시 `textureManager.reset()`을 호출해 null로 만들면, 이후 bind group 생성 시 해당 바인딩들이 완전히 빠지게 됩니다. WebGPU는 레이아웃에 선언된 모든 바인딩이 bind group에 존재해야 유효한 draw를 허용합니다.

**수정**: `generateFallback()` 메서드를 신규 추가. 1×1 픽셀의 최소 `Texture2DArray`(4레이어)를 CPU에서 직접 생성하여 항상 유효한 바인딩을 보장합니다.

| 텍스처 | 픽셀 값 | 의미 |
| ------ | ------ | ------ |
| Albedo Array | (128, 128, 128, 255) | 중간 회색 |
| Normal Array | (128, 128, 255, 255) | 평탄한 법선 (0, 0, 1) |

```cpp
// createTextureManager() — 수정 후
if (!textureManager->generate()) {
    LOG_WARN("Renderer") << "Procedural generation failed, creating fallback textures";
    if (!textureManager->generateFallback()) {
        LOG_ERROR("Renderer") << "Fallback also failed — texture bindings will be missing";
        textureManager.reset();
    }
}
```

**수정 파일**: `src/rendering/TextureManager.hpp`, `src/rendering/TextureManager.cpp`, `src/rendering/Renderer.cpp`

### 4. 그림자 형태/위치/크기 오류 — ObjectData 구조체 크기 불일치

**증상**: 그림자의 형태가 건물 실루엣과 전혀 다르고, 위치와 크기도 잘못됨

**원인**: `shaders/shadow.wgsl`의 `ObjectData` 구조체가 C++ 정의(144 bytes)와 달리 128 bytes였습니다. 마지막 필드 `texParams: vec4<f32>` (16 bytes)가 누락되어, SSBO 접근 시 모든 인스턴스의 `worldMatrix`를 잘못된 오프셋에서 읽었습니다.

**C++ ObjectData 구조 (144 bytes)**:
```
worldMatrix       64 bytes  (mat4x4)
boundingBoxMin    16 bytes  (vec4)
boundingBoxMax    16 bytes  (vec4)
colorAndMetallic  16 bytes  (vec4)
roughnessAOPad    16 bytes  (vec4)
texParams         16 bytes  (vec4)  ← 누락되어 있었음
```

**수정**: `shadow.wgsl`의 `ObjectData`에 `texParams: vec4<f32>` 필드 추가.

```wgsl
// 수정 후 shadow.wgsl ObjectData (C++ 144 bytes와 일치)
struct ObjectData {
    worldMatrix: mat4x4<f32>,
    boundingBoxMin: vec4<f32>,
    boundingBoxMax: vec4<f32>,
    colorAndMetallic: vec4<f32>,
    roughnessAOPad: vec4<f32>,
    texParams: vec4<f32>,   // 추가 — C++ ObjectData 144 bytes 일치
}
```

**수정 파일**: `shaders/shadow.wgsl`

### 5. 그림자 형태 오류 — ShadowRenderer 버텍스 스트라이드 불일치

**증상**: 그림자 형태가 건물과 전혀 다른 이상한 삼각형 형태로 출력됨 (4번과 복합적으로 작용)

**원인**: `ShadowRenderer.cpp`의 버텍스 레이아웃 스트라이드가 32 bytes로 하드코딩되어 있었습니다. 그러나 `VIS-7f`에서 `Vertex` 구조체에 `glm::vec4 tangent` (16 bytes)가 추가되어 실제 크기는 48 bytes입니다. 스트라이드가 잘못되면 GPU가 버텍스 데이터를 잘못된 위치에서 읽어 정점 좌표가 완전히 틀립니다.

**Vertex 구조체 (48 bytes)**:
```
pos       12 bytes  (vec3)
normal    12 bytes  (vec3)
texCoord   8 bytes  (vec2)
tangent   16 bytes  (vec4)  ← VIS-7f 추가, ShadowRenderer 반영 누락
```

**수정**: 하드코딩된 32 bytes → `sizeof(Vertex)`, 오프셋도 `offsetof()` 매크로 사용으로 변경. 또한 `#include "src/utils/Vertex.hpp"` 추가.

```cpp
// 이전 (하드코딩 — tangent 필드 무시)
vertexLayout.stride = sizeof(float) * 8;  // 32 bytes
vertexLayout.attributes = {
    rhi::VertexAttribute(0, 0, rhi::TextureFormat::RGB32Float, 0),
    rhi::VertexAttribute(1, 0, rhi::TextureFormat::RGB32Float, sizeof(float) * 3),
    rhi::VertexAttribute(2, 0, rhi::TextureFormat::RG32Float,  sizeof(float) * 6)
};

// 이후 (구조체 기반 — 자동으로 올바른 크기/오프셋)
vertexLayout.stride = sizeof(Vertex);     // 48 bytes
vertexLayout.attributes = {
    rhi::VertexAttribute(0, 0, rhi::TextureFormat::RGB32Float, offsetof(Vertex, pos)),
    rhi::VertexAttribute(1, 0, rhi::TextureFormat::RGB32Float, offsetof(Vertex, normal)),
    rhi::VertexAttribute(2, 0, rhi::TextureFormat::RG32Float,  offsetof(Vertex, texCoord))
};
```

**수정 파일**: `src/rendering/ShadowRenderer.cpp`

### 6. 그림자 피터패닝(Peter Panning) — Bias 과다

**증상**: 건물과 그림자 사이에 눈에 띄는 간격 발생 (그림자가 건물에서 떨어져 보임)

**원인 1**: `building.wgsl`에서 `let bias = ubo.shadowBias * 0.01`로 곱해지고 있어 실질 bias가 0에 수렴 → 이를 수정하면서 반대로 bias가 너무 커짐.

**원인 2**: Shadow frustum이 near=0.1, far=2100으로 매우 넓은 깊이 범위를 가졌습니다. NDC 단위의 bias는 실제 월드 공간 오프셋으로 환산되므로:
- `shadowBias=0.008, far-near=2100` → 실제 오프셋 ≈ 16.8m (과도한 피터패닝)

**수정 내용**:

1. Shadow frustum near/far 조정으로 깊이 범위 축소:
   ```cpp
   // 이전: near=0.1, far=lightDistance*2 (≈2100)
   // 이후: 씬 실제 범위에 맞게 타이트하게 설정
   float nearPlane = std::max(1.0f, lightDistance - sceneRadius - 300.0f);  // ≈400
   float farPlane  = lightDistance + sceneRadius + 300.0f;                  // ≈1700
   // 깊이 범위 2100 → 1300, bias 실효값 약 38% 감소
   ```

2. `Renderer.hpp`에서 `shadowBias` 기본값 축소: `0.008f` → `0.0015f`

3. `building.wgsl`에서 slope-scaled bias 적용 (경사각이 클수록 bias 증가, 최대 2배):
   ```wgsl
   // 이전: let bias = ubo.shadowBias * 0.01;  (사실상 0)
   // 이후: slope-scaled, 최대 2× 제한
   let cosTheta = clamp(dot(normal, lightDir), 0.0, 1.0);
   let bias = ubo.shadowBias * mix(2.0, 1.0, cosTheta);
   ```

**수정 파일**: `shaders/building.wgsl`, `src/rendering/ShadowRenderer.cpp`, `src/rendering/Renderer.hpp`
